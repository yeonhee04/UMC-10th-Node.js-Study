### ✅ 데이터베이스 정규화 (Database Normalization)

: 데이터의 중복을 최소화하고, 무결성(Integrity)을 보장하기 위해 테이블을 구조화하는 프로세스

- **제1정규형 (1NF):** 모든 도메인이 **원자값(Atomic Value)**으로만 구성되어야 함.
- **제2정규형 (2NF):** 부분 함수적 종속성 제거하기. (기본키의 일부분에만 종속되는 속성 분리)
- **제3정규형 (3NF):** 이행적 함수적 종속성 제거하기. (기본키가 아닌 속성 간의 종속성 분리)

**(장점)** 데이터 중복 제거로 저장 공간 절약, 데이터 삽입/수정/삭제 시 이상 현상(Anomaly) 방지.  
**(단점)** 테이블 분리로 인해 데이터를 조회할 때 **JOIN** 연산이 많아져 성능이 저하될 수 있음.

---

### ✅ 인덱스 (Index)

: 데이터베이스 테이블의 검색 속도를 높이기 위해 별도로 관리하는 자료구조

- **B-Tree 구조:** 대부분의 RDBMS는 밸런스드 트리(B-Tree) 구조로 균형 잡힌 탐색 시간을 보장함.
- **기본키(PK):** 자동으로 인덱스가 생성되는 경우가 많음.

**(장점)** `SELECT` 질의 시 검색 속도와 시스템의 전반적인 성능을 획기적으로 향상시킴.  
**(단점)** `INSERT`, `UPDATE`, `DELETE` 작업 시 인덱스를 매번 갱신해야 하므로 쓰기 성능이 떨어지고, 인덱스를 저장하기 위한 추가적인 물리 공간이 필요함.

---

### ✅ ORM vs. Raw SQL

: 애플리케이션에서 데이터베이스에 접근하는 두 가지 주요 방식

| 구분         | ORM (Object-Relational Mapping)                    | Raw SQL                             |
| :----------- | :------------------------------------------------- | :---------------------------------- |
| **정의**     | 객체와 DB 테이블을 매핑<br>(TypeORM, Hibernate 등) | SQL 쿼리를 개발자가 직접 작성       |
| **생산성**   | 매우 높음 (코드 중심의 개발 가능)                  | 낮음 (반복적인 쿼리 작성 필요)      |
| **가독성**   | 객체 지향적 코드로 직관적임                        | 복잡한 쿼리의 경우 코드 가독성 저하 |
| **제어권**   | 세밀한 최적화가 어려울 수 있음                     | DB의 모든 기능을 100% 활용 가능     |
| **유지보수** | DB 엔진 변경 시 대응이 쉬움                        | DB 종속성이 높아 변경 시 비용 발생  |
